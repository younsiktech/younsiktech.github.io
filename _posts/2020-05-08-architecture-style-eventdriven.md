## 이벤트 기반 아키텍처 (Event-Driven Architecture)

![001](/images/20200508/event-driven.png)

### 요약

이벤트 기반 아키텍처는 이벤트를 생성하는 이벤트 생산자(Producer)와 이벤트를 수신 대기하는 이벤트 소비자(Consumer)로 구성된다. 생산자는 소비자와 독립적 관계이며, 각 소비자는 서로 독립적 관계이다. 이벤트는 거의 실시간으로 전달되므로 이벤트가 발생하는 즉시 소비자가 이벤트에 응답할 수 있다.

생산자와 소비자가 분리되므로 생산자는 수신 대기 중인 소비자를 알 수 없다. 소비자도 서로 분리되며 각 소비자에게 모든 이벤트가 표시된다.

이는 소비자가 큐에서 메시지를 끌어오고 오류가 없다고 가정할 경우 메시지가 한 번만 처리되는 경쟁 소비자 패턴과 다르다.

실제로는 소비자가 시스템의 단일 실패 지점이 되지 않도록 한 소비자의 여러 인스턴스가 있는 것이 일반적이다.

이벤트의 볼륨 및 빈도를 처리하기 위해 여러 인스턴스가 필요할 수도 있다.

또한 단일 소비자가 여러 스레드의 이벤트를 처리할 수 있다. 이렇게 하는 경우 이벤트를 순서대로 처리하거나 정확히 한 번의 의미 체계를 요구 하는 경우 문제가 발생할 수 있다.

### 사용하는 경우

- 여러 하위 시스템이 동일한 이벤트를 처리해야 하는 경우
- 최소 시간 지연의 실시간 처리
- 패턴 일치 또는 일정 기간의 집계와 같은 복합 이벤트 처리
- 높은 볼륨 및 높은 데이터 개발속도(예: IoT)

### 장점

IoT 솔루션과 같은 대규모 데이터를 수집하여 처리하고 대기 시간이 매우 짧은 애플리케이션에는 이벤트 기반 아키텍처를 사용한다. (IoT와 같은 일부 시스템에서는 매우 높은 볼륨으로 이벤트를 수집해야 한다)

이 스타일은 여러 하위 시스템이 동일한 이벤트 데이터에서 다양한 종류의 처리 작업을 수행해야 하는 경우도 유용하다.

- 생산자와 소비자가 분리됩니다.
- 지점 간 통합이 없습니다. 시스템에 새 소비자를 쉽게 추가할 수 있습니다.
- 이벤트가 도착하는 즉시 소비자가 이벤트에 응답할 수 있습니다.
- 확장성이 있고 배포 가능합니다.
- 하위 시스템에서 이벤트 스트림을 독립적으로 확인할 수 있습니다.

### 고려할 점

- 배달 보장
  - 일부 시스템, 특히 IoT 시나리오에서는 이벤트가 배달되도록 보장하는 것이 중요하다.
- 이벤트를 순서대로 또는 한 번만 처리
  - 복원 및 확장성을 위해 일반적으로 각 소비자 유형이 여러 인스턴스에서 실행된다.
  - 이벤트가 소비자 유형 내에서 순서대로 처리되어야 하거나 처리 논리가 비멱등적인 경우 이것이 문제가 될 수 있다.

### 아키텍처 모델별 차이

이벤트 기반 아키텍처는 게시자/구독자(pub-sub) 모델 또는 이벤트 스트림 모델을 사용할 수 있다.

- 게시자/구독자 모델
  - 메시징 인프라에서 구독을 추적한다.
  - 이벤트가 게시되면 각 구독자에게 이벤트를 보낸다.
  - 이벤트를 받은 후에는 재생할 수 없으며 새 구독자에게 이벤트가 표시되지 않는다.

- 이벤트 스트리밍
  - 이벤트가 로그에 기록된다.
  - 이벤트가 파티션 내에 엄격하게 정렬되며 지속 가능하다.
  - 클라이언트는 스트림을 구독하지 않고, 대신 스트림의 일부에서 읽을 수 있다.
  - 클라이언트가 스트림에서 해당 위치를 진행해야 한다.
  - 즉, 클라이언트가 언제든지 연결하고 이벤트를 재생할 수 있다.

### 소비자의 변형

서비스에 따라 소비자 측면에서 다음과 같은 몇 가지 일반적인 변형이 있을 수 있다.

- 단순 이벤트 처리
  - 이벤트가 즉시 소비자에서 작업을 트리거합니다.
  - 예를 들어 서비스 버스 토픽에 메시지를 게시할 때마다 함수가 실행되도록 서비스 버스 트리거와 함께 사용할 수 있다.
  - ex) 서버리스 서비스
    - AWS : Lambda
    - Azure : Functions
    - Google Cloud : Cloud Functions

- 복합 이벤트 처리
  - 소비자가 이벤트 데이터에서 패턴을 찾으면서 일련의 이벤트를 처리합니다.
  - 예를 들어 포함된 디바이스에서 일정 기간 동안 읽은 값을 집계하고 이동 평균이 특정 임계값을 초과하면 알림을 생성할 수 있습니다.
  - ex) 스트리밍 데이터 파이프라인 서비스
    - AWS : Kinesis
    - Azure : Stream Analystics
    - Google Cloud : Cloud Dataflow
    - Apache Storm 등등

- 이벤트 스트림 처리
  - 데이터 스트리밍 플랫폼을 파이프라인으로 사용하여 이벤트를 수집하고 스트림 프로세서에 공급한다.
  - 스트림 프로세서는 스트림을 처리하거나 변환한다.
  - 애플리케이션의 각 하위 시스템에 사용되는 여러 스트림 프로세서가 있을 수 있다. (IoT 워크로드에 적합)
  - ex) 데이터 스트리밍 플랫폼
    - Apache Kafka 등등
