## MYSQL 컬럼 타입

### 문자 데이터

```
1. CHAR(n), VARCHAR(n)에서 n은 byte가 아닌 글자 수를 의미함 (mysql 4.1 이후부터, 이전에는 byte)
2. UTF-8로 변환시 한글과 영어 길이가 같아진다
3. BINARY 키워드가 주어지지 않으면 대소문자 구분 없이 검색한다
4. MySQL에서 하나의 레코드(row)에서 TEXT, BLOB을 제외한 컬럼의 전체 크기는 64kb를 넘을 수 없다
5. UTF-8문자는 최대 3byte를 사용하므로 실제 저장 가능한 글자 수를 계산해야한다. (한글은 2byte)
```

#### CHAR

- 고정 길이 문자열 저장
- 최대 길이 (255)
- 크기보다 작은 문자열 저장시 뒷부분은 공백으로 채워짐 : 공백채움비교, 공간의 낭비 발생
  - 데이타를 가져올때는 공백을 제거하고 가져옴
- INSERT 되는 길이가 동일할때 사용한다
- 자주 업데이트 되는 경우 고정 길이가 좋다
- 검색 속도가 빠르다

#### VARCHAR

- 가변 길이 문자열 저장
- 최대 길이 (65535) : 5.0.3부터 바뀜, 이전에는 255
- 길이에 대한 정보를 저장하는 byte가 필요하다
  - 255이하는 1byte, 이상은 2byte 추가 필요
- INSERT 되는 길이가 다를때 사용한다
- 버젼별로 공백 처리가 다르다
  - 5.0.3 이후 : 뒤따르는 공백 제거하지 않고 보관하고 읽어들일 때도 보존
  - 5.0.3 이전 : 뒤따르는 공백 제거하고 읽을 때 제거된 데이터 읽음
- 파편화가 발 생 할 수 있다.
  - 기존 보다 큰 데이터 저장하려면 새로운 저장영역에 새로 할당하기 때문에 데이터 파편화가 발생
  - 특히, CHAR와 VARCHAR가 한 row에 섞여 있으면 파편화가 발생한다
    - 이를 막으려면 ROW_FORMAT=FIXED를 사용해서 VARCHAR를 CHAR처럼 사용한다
    - 데이터 용량은 증가하나 파편화로 따른 성능저하는 막을 수 있다
- 4자 이하의 VARCHAR는 CHAR로 자동 변환된다
  - 반대로 text, blob같은 가변 길이 데이터가 있는 CHAR는 3자 이상이면 VARCHAR로 바뀐다
- DESC {테이블명} 명령어를 통해서 실제 변환된 타입을 알 수 있다

#### TEXT

- 커다란 텍스트 데이터를 저장할 때 사용한다
- 문자 셋을 가지고 있어서 문자 셋의 collation 기반으로 값을 저장하고 비교한다
- VARCHAR와 다른 점은 색인화가 불가능 하다
  - 따라서 검색할 때 전문 검색을 하여 퍼포먼스가 좋지 않다
- 종류
  - TINYTEXT
    - 잘 사용하지 않는다. 차라리 데이터 무결성 측면에서 VARCHAR를 사용하는게 낫다.
  - TEXT
  - MEDIUMTEXT
  - LONGTEXT
- 보관 용도로 사용할 때 좋다
- 데이터 검색 속도에 신경 쓰지 않는다
- 검색할 때 LIKE 연산자를 사용하여 색인 생성이 도움이 되지 않는다
- 데이터 길이의 한계를 예측할 수 없다
- 가능하면 별도 테이블로 분리하여 left join으로 가져오는게 좋다

#### BLOB

- 바이너리 데이터를 저장할 때 사용한다
- 문자셋을 가지고 있지 않아서 정렬시 방이트 숫자 기반으로 정렬한다
- 종류
  - TINYBLOB
  - BLOB
    - 가능하면 파일 경로를 VARCHAR로 저장하고 쓰자
  - MEDIUMBLOB
  - LONGBLOB

#### 저장 방법

| 타입     | 0   | 1   | 2   | 3   | 4   | 5   | 6   | 7   |
|:--------|:---:|:---:|:---:|:---:|:---:|:---:|:---:|----:|
| CHAR    |  T  |  E  |  C  |  H  | 공백 | 공백 | 공백 | 공백 |
| VARCHAR |  T  |  E  |  C  |  H  | 길이 |     |     |     |

#### 주의할점

- 비교방법 (#을 공백으로 생각)
  - CHAR(8)에서 `AA` = `AA##` 를 비교하면 `AA######` == `AA######`가 되어 같다는 결과
  - VARCHAR(8)에서 `AA` == `AA##`를 비교하면 다르다는 결과
- Strict SQL Mode
  - 활성 여부에 따라 공백이 아닌 문자열을 자르거나 값을 추가 시에 에러 발생 할 수 있다.

    | 값 | CHAR(4) | 필요 저장 크기 | VARCHAR(4) | 필요 저장 크기 |
    |:---|:---:|:---:|:---:|---:|
    | '' | '####' | 4byte | '' | 1byte |
    | 'a' | 'a###' | 4byte | 'a' | 2byte |
    | 'ab' | 'ab##' | 4byte | 'ab' | 3byte |
    | 'abc' | 'abc#' | 4byte | 'abc' | 4byte |
    | 'abcd' | 'abcd' | 4byte | 'abcd' | 5byte |
    | 'abcdefg' | 'abcd' | 4byte | 'abcd' | 5byte |

  - 초과 문자가 공백문자인 경우
    - CHAR는 해당 길이를 초과하면 SQL Mode 상관없이 잘린후 저장되고 경고가 발생하지 않는다
    - VARCHAR는 해당 길이를 초과하면 SQL Mode 상관없이 잘린후 저장되고 경고가 발생한다
  - 초과 문자가 공백문자가 아닌 경우
    - 길이를 초과하면 저장되지 않고 에러를 발생한다
    - 마지막 열처럼 저장하려면 Strict SQL Mode를 사용하지 않으면 된다

### 정수형 데이터

#### INT

- INT(n)의 n은 ZeroFill을 의미함
  - value = 2335 일 경우
    - INT(3) : 2335
    - INT(5) : 02335
    - INT(7) : 0002335
- 종류
  - TINYINT
  - SMALLINT
  - MEDIUMINT
  - INT
  - BIGINT

### 실수형 데이터

#### DECIMAL(M,N)

- 정확한 소수를 표현할 때 사용한다
- FLOAT, DOUBLE과 같은 부동 소수점 타입은 유효범위 이외의 값은 보장되지 않는다
  - 따라서 고정된 소수점까지 관리할때 DECIMAL을 사용한다
- M은 총 자릿수, N은 소수부 최대 자릿수를 나타낸다

### 상수형 데이터

#### ENUM

- radio 버튼과 비슷하다
- 최대 65535개의 값이 들어갈 수 있다. (0은 불가, NULL은 가능)
- 단일 값만 저장된다

#### SET
- checkbox 버튼과 비슷하다
- 최대 64개의 값이 들어갈 수 있다. (ENUM과 동일)
- INSERT 시에 자동으로 정렬 및 중복시 하나만 들어감
- 비트 단위 연산이 가능하다
- chmod처럼 스위치 형태 구성이 가능하다 (복수 선택 + 비트 단위 연산)
  
    | 옵션   | 비회원 글쓰기(W) | 비회원 글보기(V) | 비회원 리플(R) | 비회원 답글달기(A) |
    |:------|:-------------:|:------------:|:------------:|---------------:|
    | 10진수 | 1 | 2 | 4 | 8 |
    | 2진수  | 0001 | 0010 | 0100 | 1000 |
  
  - 위의 형태를 SET을 이용하여 R W V A 등의 문자형태로 구성할 수 있다
